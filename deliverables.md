1. Project Description
Your group will design and execute an end-to-end data analytics project on a topic of your
choice. The project should incorporate analytical techniques covered throughout this course,
selecting those that best suit your chosen problem. The goal is not simply to produce a
working model, but to demonstrate a complete analytical workflow: from problem formulation
and data acquisition through analysis, interpretation, and communication of actionable
insights.
This is your opportunity to apply the skills you have developed across the course in a
realistic, self-directed context. You will be assessed not only on technical execution, but on
the quality of your analytical reasoning, the clarity of your documentation, and the strength of
your final communication.
1.1 Learning Objectives
• Formulate a well-scoped analytical question from a real-world problem
• Acquire, clean, and integrate data from multiple sources
• Apply multiple analytical techniques from the course in a coherent pipeline
• Interpret results critically, documenting decisions and limitations
• Communicate findings effectively through a written report and oral presentation
• Collaborate productively in a team, managing roles and contributions
1.2 Scope and Requirements
Your project must satisfy all of the following requirements:
• Original topic: Each group selects its own topic and formulates its own research
question. Topics must be approved by the instructor within the first two weeks.
• Real-world data: The project must use real, publicly available datasets. Toy or
synthetic datasets are not acceptable as the primary data source.
• Techniques from the course: Your project should incorporate analytical techniques
covered during the course. The more techniques you meaningfully integrate, the
stronger your project will be. These include, but are not limited to: data collection via
APIs or web scraping, time series analysis, geospatial analysis, machine learning
(classification, regression, clustering), and neural networks (ANN or CNN). Choose
the techniques that best fit your problem rather than forcing all of them in.
• Analytical depth: Each technique must be applied with sufficient depth to
demonstrate genuine understanding. Superficial application of many techniques is
less valuable than thorough application of fewer.
• Reproducibility: All code must be executable and produce the results described in
the report. Notebooks should run from top to bottom without errors

2.1 Technical Report (PDF, max 20 pages)

A structured report containing the following sections:
• Executive Summary (1 page): A concise overview of the problem, approach, key
findings, and recommendations. Written so that a non-technical reader can
understand the value of the work.
• Introduction and Problem Statement: What question are you answering and why
does it matter? Include context and motivation.
• Data Description: Sources, acquisition method, volume, time range, geographic
scope. Describe any quality issues encountered and how they were addressed.
• Methodology: For each analytical technique applied, explain why it was chosen,
how it was configured, and what alternatives were considered. Document key
decisions and trade-offs.
• Results and Analysis: Present findings with appropriate visualizations. Interpret
what the results mean in the context of your question. Do not simply report metrics;
explain what they tell you.
• Limitations and Future Work: What can your analysis not answer? What would you
do differently with more time or resources?
• Conclusions and Recommendations: Actionable, evidence-based
recommendations tied directly to your analysis.
• References: Cite all external sources, datasets, and tools used.
2.2 Jupyter Notebook(s)
Fully executable notebooks containing all code used in the project. Requirements:
• Notebooks must run sequentially from top to bottom without errors
• Include markdown cells explaining the purpose of each code block and interpreting
outputs
• All data transformations, model configurations, and parameter choices must be
documented
• If the project spans multiple notebooks, include a README explaining the execution
order
