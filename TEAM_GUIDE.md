## What Are We Building?

We're building a system that **predicts how congested Madrid-Barajas airport will be in the next 1-3 hours**, using real public data from flight trackers, weather APIs, and holiday calendars.

Think of it like a weather forecast, but for airport traffic. Instead of predicting rain, we predict: *"Will the airport be unusually busy at 4pm today?"*

### The Core Idea

1. We download **historical flight data** (how many planes arrived/departed each hour for the past year)
2. We add **weather data** (temperature, wind, rain) and **calendar info** (holidays, weekends)
3. We compute an **Airport Congestion Pressure Score (ACPS)** - our own metric that measures how busy the airport is compared to what's normal for that time
4. We train **machine learning models** to predict that score 1 hour and 3 hours into the future
5. We visualize everything with **maps** showing where planes cluster around the airport

## Our Target: The ACPS Score

**Airport Congestion Pressure Score** - this is the thing we're predicting.

For each hour, we:

1. Count total flight movements (arrivals + departures)
2. Compare that to the typical number for that same hour and day-of-week
3. Combine both into a single score

Then we classify each hour as:

- **Low congestion** (below 60th percentile - normal)
- **Medium congestion** (60th-85th percentile - busy)
- **High congestion** (above 85th percentile - very busy, potential disruptions)

This is transparent and defensible. We're not claiming to predict actual delays (we don't have that data). We're predicting *relative busyness compared to what's normal*.

## C2 - Feature Engineering & Forecasting

**Your job:** Take C1's raw data and turn it into model-ready features, then build the first models.

### Your files (already scaffolded with function signatures)

All stubs are in `src/features/` and `src/modeling/`. Each file has:

- The exact function signature you need to implement
- A docstring explaining what the function should do
- Import statements already set up

### Step-by-step instructions

**Step 1: Hourly aggregation** (`src/features/build_hourly_movements.py`)

- Read arrivals parquet, count arrivals per hour
- Read departures parquet, count departures per hour
- Merge into one table: timestamp | arrivals | departures | total_movements
- Save to `data/intermediate/hourly_movements.parquet`

**Step 2: ACPS target** (`src/features/build_model_table.py`)

- For each hour, compute the baseline (median movements for that hour-of-day and day-of-week)
- Compute pressure_ratio = movements / baseline
- ACPS = 0.6 *standardized(movements) + 0.4* standardized(pressure_ratio)
- Classify: Low (<60th pctl), Medium (60-85th), High (>85th)

**Step 3: Calendar features** (`src/features/build_calendar_features.py`)

- hour (0-23), day_of_week (0=Mon), is_weekend, month, quarter
- is_holiday, is_pre_holiday (day before), is_post_holiday (day after)
- is_bridge_day (workday between holiday and weekend)

**Step 4: Weather features** (`src/features/build_weather_features.py`)

- Keep all raw weather variables
- Encode wind direction as sin/cos (wind_dir_sin, wind_dir_cos)
- Add rain_flag (1 if precipitation > 0)
- Add severe_weather_flag (1 if weather_code indicates storms/heavy rain)

**Step 5: Lag & rolling features** (`src/features/build_model_table.py`)

- Lags of ACPS: 1h, 2h, 3h, 6h, 12h, 24h, 48h, 168h (same hour last week)
- Rolling mean: 3h, 6h, 24h
- Rolling std: 24h
- Ratio to last week same hour

**Step 6: Assemble & split** (`src/features/build_model_table.py`)

- Merge all features on timestamp
- Split chronologically: first 70% train, next 15% validation, last 15% test
- Save to `data/processed/train.parquet`, `valid.parquet`, `test.parquet`

**Step 7: Baselines** (`src/modeling/baselines.py`)

- Previous hour (just use last known value)
- Same hour yesterday
- Same hour last week
- For classification: majority class

**Step 8: SARIMAX** (`src/modeling/sarimax_model.py`)

- Classical time series model with weather as exogenous variables
- Don't over-tune. Document parameter choices.

**Step 9: Notebooks 04, 05, 07, 08**

- Use the template in `.claude/tools/notebook_template.md`
- Each notebook must explain WHAT you did and WHY

### Your output

When done, these files must exist:

- `data/processed/train.parquet`, `valid.parquet`, `test.parquet`
- `outputs/tables/baseline_comparison.csv`
- Notebooks 04, 05, 07, 08 with visible outputs
